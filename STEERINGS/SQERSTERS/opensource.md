# Open Source Contribution Rules

These rules apply ONLY when working on an open source project where the user is a solo contributor. If the project is closed-source (private, proprietary, or internal), these rules do not apply — follow standard development workflow instead.
## NEVER
- Modify docs (README, CHANGELOG, TODO, etc.) — the upstream project owns those
- Modify any code without first identifying the root cause and getting explicit user approval

## Workflow
1. **Investigate** — Find the source of the problem first
2. **Explain** — Present findings and proposed fix to the user
3. **Ask permission** — Wait for explicit approval before making changes
4. **Fix** — Apply the minimal, targeted fix
5. **Test** — Build/run tests to verify the fix works
6. **PR** — Create a pull request with a clean, human-written style:
   - Use a short dash `-` for list items, not big bullet bars
   - Keep the tone concise and natural, not AI-generated sounding
   - If no fork exists, fork the repo first, then open the PR
7. **Code Styles** — Follow existing code style, patterns, and conventions strictly.
8. **Never copy paste codes from others project** — don't copy paste codes from others project. Respect Licenses, Respect Contributing.md.
